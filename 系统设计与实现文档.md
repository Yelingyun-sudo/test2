# 第一章 系统整体架构

## 1.1 架构概览

如图 1-1 所示，系统的分层设计体现了清晰的职责边界。前端展示层采用 Next.js 框架构建，为用户提供数据可视化和任务管理界面；API 网关层基于 FastAPI 实现，负责请求路由、身份认证和数据持久化；任务调度层是系统的核心调度中枢，管理任务的全生命周期；AI Agent 层实现智能化的浏览器操作决策；浏览器驱动层则负责与真实浏览器的底层交互。这种分层架构使得各组件可以独立演进，例如 AI Agent 层可以从 OpenAI 模型切换到其他 LLM 提供商，而无需修改上层代码。

## 1.2 核心模块组成

系统由四个核心模块组成，各模块之间通过定义良好的接口进行协作，共同完成网站分析任务。

### 1.2.1 FastAPI 后端服务

FastAPI 后端服务是系统的业务逻辑处理中心，采用 Python 3.12 开发，遵循异步编程模型。服务层主要包含以下组件：

- **路由层（Routers）**：定义 RESTful API 端点，包括任务管理、订阅任务、取证任务、支付任务等路由模块。每个路由模块负责一类业务功能的 HTTP 接口暴露。

- **数据模型层（Models）**：使用 SQLAlchemy ORM 定义数据实体，主要包括任务模型（Task）、订阅任务模型（SubscriptionTask）、取证任务模型（EvidenceTask）、支付任务模型（PaymentTask）等。模型之间通过外键关联，形成完整的数据关系网络。

- **数据访问层（Repositories）**：实现数据访问对象（DAO）模式，将数据库操作逻辑从业务逻辑中分离，提供统一的数据CRUD接口。

- **模式定义层（Schemas）**：使用 Pydantic 定义请求和响应的数据校验模式，确保接口数据的类型安全和自动文档生成。

### 1.2.2 Next.js 前端控制台

前端控制台采用 Next.js 14 的 App Router 模式构建，结合 React 18 和 TypeScript 技术栈，为用户提供直观的操作界面：

- **页面路由**：基于文件系统的路由结构，包含概览页（Dashboard）、订阅管理页、取证管理页、支付管理页等功能模块。

- **组件体系**：采用组件化设计，公共组件（common）提供复用的 UI 元素，业务组件按功能域组织（subscription、evidence、payment）。

- **样式方案**：使用 Tailwind CSS 进行原子化样式开发，结合 shadcn/ui 组件库，保持界面的一致性和美观性。

- **状态管理**：使用 React Hook Form 处理表单状态，Zod 进行表单校验，实现类型安全的状态管理。

### 1.2.3 Kafka 消息队列

Kafka 消息队列承担系统异步通信的基础设施角色，实现任务导入和结果上报的解耦：

- **任务导入 Topic**：接收外部系统提交的任务指令，支持订阅任务、取证任务、支付任务三种类型。任务导入器（Task Importer）消费该 Topic，将任务持久化到 SQLite 数据库。

- **结果上报 Topic**：分为成功上报 Topic 和失败上报 Topic，任务执行完成后，报告器将执行结果（包括状态、耗时、输出、错误类型等）发送到对应 Topic，供下游系统消费。

- **消费者组管理**：采用消费者组模式实现任务的负载均衡和容错处理，确保高可用性。

### 1.2.4 SQLite 数据存储

SQLite 作为系统的持久化存储方案，满足中小型部署场景的需求：

- **数据库文件位置**：默认位于 `backend/db/wa.db`，通过环境变量可配置。

- **核心数据表**：包括任务表（存储任务元数据）、订阅任务表（扩展订阅专用字段）、取证任务表（扩展取证专用字段）、支付任务表（扩展支付专用字段）。

- **数据关系**：任务表作为基表，通过一对多关系与各类子任务表关联，支持灵活的任务类型扩展。

**图 1-2 端到端数据流转图**

数据流转的完整过程如下：首先，外部任务源（可以是上游业务系统、管理后台的手动提交或批量导入脚本）将任务指令以 JSON 格式发送到 Kafka 的任务导入 Topic。任务导入器作为消费者持续监听该 Topic，接收到消息后执行解析和校验，将合法的的任务数据写入 SQLite 数据库，并初始化任务状态为 PENDING（待执行）。

任务执行器通过轮询或事件触发机制获取待执行的任务，将其状态更新为 RUNNING（执行中）。在执行阶段，系统会启动浏览器实例，调用 AI Agent 进行智能化的页面操作。执行完成后，任务状态被标记为 COMPLETED（成功完成）或 FAILED（执行失败），同时将执行结果（包括结构化输出、截图路径、视频路径、LLM Token 消耗等）持久化到数据库。

任务报告器负责将执行结果发送到 Kafka 的结果上报 Topic。根据执行状态的不同，结果会被发送到成功上报 Topic 或失败上报 Topic，供下游的数据分析系统、监控告警系统或业务系统消费处理。整个数据流转过程实现了异步解耦，即使下游系统暂时不可用，也不会影响主流程的执行。

# 第二章 AI Agent 多代理架构

## 2.1 代理协作模型

## 2.2 工具链与安全控制

系统的工具链设计遵循"最小权限原则"，通过多层安全机制确保 AI Agent 的操作可控可审计。

### 2.2.1 Playwright MCP Server 集成

Playwright MCP Server 是系统的浏览器自动化基础设施，基于 Model Context Protocol（MCP）协议实现与 AI Agent 的通信。MCP 协议定义了标准化的工具调用接口，使得 AI Agent 可以通过统一的 API 控制浏览器行为。

系统通过 `AutoSwitchingPlaywrightServer` 类对 MCP Server 进行封装，提供以下增强功能：

- **标签页自动切换**：监控浏览器标签页的变化，自动将控制权切换到最新打开或当前激活的标签页，解决多标签场景下的上下文管理问题。

- **优雅关闭与资源清理**：在任务结束时执行浏览器的优雅关闭流程，包括调用 `browser_close`、逐个关闭标签页、清理孤儿 Chrome 进程，避免资源泄漏。

- **视频录制集成**：自动配置 Playwright 的视频录制参数，将执行过程录制为 WebM 格式的视频文件，用于后续的审计和故障排查。

### 2.2.3 邮箱验证码获取工具

注册代理需要处理邮箱验证码验证流程。系统通过 IMAP 协议与邮件服务器通信，实现验证码的自动获取。邮箱验证码获取工具的核心逻辑包括：

- **异步 IMAP 连接**：使用 `aioimaplib` 库建立异步 IMAP 连接，避免阻塞事件循环。

- **重试机制**：配置最大重试次数（默认 6 次）和重试间隔（默认 15 秒），在验证码邮件未到达时进行轮询。

- **邮件过滤**：基于邮件主题关键词（如"验证码"、"verification"）和发送时间（只处理最近 2 分钟内的邮件）进行精确过滤。

- **验证码提取**：使用正则表达式从邮件正文中提取数字验证码，支持多位数字和字母数字混合的验证码格式。

## 2.3 代理编排实现

代理编排是多代理架构的核心，负责协调各子代理的执行顺序和数据传递。

### 2.3.1 agent_factory.py 代理构建逻辑

`agent_factory.py` 模块实现了工厂模式，封装了各代理的创建逻辑。工厂函数负责：

- **指令加载**：从 `backend/instructions/` 目录加载 Markdown 格式的代理指令文件，支持变量占位符替换（如 `{REGISTER_ACCOUNT}`、`{MAX_MENU_ENTRIES}`）。

- **模型配置**：从环境变量读取 LLM 模型配置（默认使用 `gpt-5.1-codex-mini`），设置结构化输出类型。

- **工具绑定**：为代理绑定必要的工具，如注册代理绑定邮箱验证码获取工具，取证代理绑定截图和报告生成工具。

- **MCP Server 关联**：将 Playwright MCP Server 实例与需要浏览器控制的代理关联。

### 2.3.2 子代理转工具（as_tool）机制

子代理通过 `as_tool()` 方法被封装为协调代理可调用的工具，这是实现代理编排的关键机制。`as_tool()` 的调用参数包括：

- **tool_name**：工具的唯一标识符，如 "login"、"register"、"extract"、"evidence"、"payment"。

- **tool_description**：工具功能描述，供协调代理理解工具用途。

- **max_turns**：代理执行的最大轮数限制，防止无限循环（如登录代理限制 20 轮，取证代理限制 100 轮）。

- **hooks**：运行时钩子，用于捕获执行日志和浏览器状态。

- **custom_output_extractor**：自定义输出提取器，将代理的结构化输出序列化为 JSON 字符串。

### 2.3.3 结构化输出解析与传递

系统采用 Pydantic 模型定义各代理的结构化输出，确保数据的类型安全和自动校验。结构化输出的处理流程如下：

1. **输出生成**：子代理执行完成后，`Runner.run()` 返回 `RunResult` 对象，其中 `final_output` 属性包含 Pydantic 模型实例。

2. **序列化**：通过 `extract_structured_output()` 函数将 Pydantic 模型序列化为 JSON 字符串，使用 `model_dump(mode='json')` 确保输出只包含定义的字段。

3. **结果汇总**：协调代理收集各子代理的执行结果，将其组织为 `operations_results` 字典，键为操作类型（如 "login"、"extract"），值为对应代理的完整输出。

4. **最终输出**：协调代理生成 `CoordinatorOutput`，包含任务整体状态（success/failed）、消息、错误类型、执行的操作列表以及各操作的详细结果。

这种设计使得下游系统可以精确地了解任务的执行细节，例如不仅知道"登录失败"，还能知道失败原因是"账号被封禁"还是"反自动化检测"，从而采取针对性的处理措施。

# 第三章 任务调度与管理系统

任务调度与管理系统是 Website Analytics 平台的执行中枢，负责任务从接收到完成的全生命周期管理。该系统采用异步事件驱动架构，通过独立运行的后台协程实现任务的导入、执行、清理和上报四大核心功能。调度层的设计理念在于将任务流与业务逻辑解耦，使得上层 AI Agent 可以专注于自动化操作本身，而无需关心任务的排队、并发控制和状态持久化等基础设施问题。

## 3.1 任务模型设计

系统中的任务被抽象为三种业务类型：订阅任务（SubscriptionTask）、取证任务（EvidenceTask）和支付任务（PaymentTask）。这三种任务类型共享同一套状态机模型，但各自承载不同的业务数据。任务状态流转遵循严格的顺序约束：从初始的 PENDING（待执行）状态开始，经过 RUNNING（执行中）状态，最终到达 SUCCESS（成功完成）或 FAILED（执行失败）状态。这种三态模型简单明了，足以覆盖任务执行的主要生命周期阶段，同时也便于监控和统计。

任务模型在数据库层面的设计体现了对业务需求的精细考量。以订阅任务为例，除了基本的 URL、账号、密码字段外，还包含状态字段（status）、执行时长（duration_seconds）、任务目录（task_dir）、执行结果（result）、失败类型（failure_type）以及 LLM Token 使用统计（llm_usage）等多个维度。特别值得注意的是，任务表设置了联合唯一约束（url + account + created_date），防止同一天内对同一网站使用相同账号的重复任务入库。LLM 使用统计以 JSON 格式存储，灵活地记录了每次任务执行的 Token 消耗明细，为后续的成本分析和模型优化提供数据支撑。任务模型的这些字段设计使得系统能够完整地追溯每次执行的上下文和结果，无论是成功的订阅地址提取，还是失败的错误归因，都被精确地记录在案。

## 3.2 调度器架构

任务调度层采用多组件协作的架构设计，核心由任务导入器（Task Importer）、任务执行器（Task Runner）、任务清理器（Task Cleaner）和任务报告器（Task Reporter）四个独立模块组成。这四个组件作为后台协程并行运行，通过数据库状态变更和消息队列实现协同工作。

任务导入器负责从 Kafka 消息队列消费任务指令。它采用独立线程运行同步的 Kafka 消费者，将轮询到的消息解析后写入数据库。导入器支持三种任务类型的自动识别：当消息包含 task_type="payment" 时创建支付任务；包含 task_type="subscription" 或有账号密码字段时创建订阅任务；仅有 URL 时创建取证任务。导入器还实现了消息过期检查机制，自动跳过超过配置时间（默认 24 小时）的旧消息，防止过期指令的执行。

任务执行器是调度层的核心执行组件，它定期轮询数据库中处于 PENDING 状态的任务，将其状态更新为 RUNNING 后调用底层的 AI Agent 执行自动化流程。执行器采用信号量（Semaphore）机制实现并发控制，通过配置 max_concurrent 参数限制同时运行的任务数量，避免浏览器实例过多导致系统资源耗尽。执行过程中，执行器捕获 Agent 返回的结构化结果，根据执行成功与否更新数据库状态，并填充相应的执行时长、结果内容和失败类型等字段。

任务清理器作为系统的容错保障机制，负责处理执行过程中可能发生的异常中断。由于浏览器自动化任务涉及复杂的网络环境和页面交互，任务可能因进程崩溃、网络断开或页面卡死而长期处于 RUNNING 状态而无法推进。清理器定期检查执行时间超过阈值（默认 15 分钟）的 RUNNING 任务，将其状态强制标记为 FAILED，并设置失败类型为 "task_cleaned"。这种机制确保了系统的自我恢复能力，防止僵尸任务占用执行资源。

任务报告器负责任务执行结果的上报。它轮询数据库中状态为 SUCCESS 或 FAILED 但 report_status 仍为 PENDING 的任务，将执行结果封装为 JSON 消息后发送到 Kafka 的相应 Topic。成功任务发送到成功上报 Topic，失败任务发送到失败上报 Topic。报告器实现了 Kafka 生产者的单例复用和连接池管理，确保高吞吐场景下的消息发送效率。报告完成后，任务的 report_status 被更新为 SUCCESS 或 FAILED，形成完整的闭环。

## 3.3 并发与资源控制

由于浏览器自动化任务属于资源密集型操作，每个任务都需要启动独立的 Chromium 浏览器实例，系统必须对并发度进行严格控制。任务执行器通过 asyncio.Semaphore 实现并发信号量控制，在初始化时根据配置参数创建具有指定容量的信号量对象。执行器在调用 Agent 前首先尝试获取信号量，当并发任务数达到上限时，后续任务将阻塞等待，直到有任务完成并释放信号量。这种机制既保证了系统不会过载，又实现了任务队列的自动缓冲。

Playwright 浏览器实例的生命周期管理与任务执行紧密绑定。每个任务在执行时创建独立的浏览器上下文（Browser Context），拥有隔离的 Cookie、缓存和本地存储，确保任务之间不会相互干扰。任务完成后，系统执行严格的资源清理流程：首先尝试优雅地关闭浏览器页面和上下文，然后关闭浏览器实例，最后强制清理可能残留的孤儿 Chrome 进程。这种分层清理策略有效地防止了浏览器进程泄漏，确保了长期运行的稳定性。

任务超时控制采用双层机制。第一层是执行器层面的整体超时，通过 asyncio.wait_for 为整个 Agent 执行流程设置超时时间（默认 10 分钟），超时后强制取消任务执行；第二层是 Playwright 层面的操作超时，为单个浏览器操作（如页面导航、元素点击）设置更短的超时时间，防止因页面卡死导致的无限等待。两层超时机制相互配合，既保护了系统资源，又为故障诊断提供了明确的超时信号。

# 第四章 浏览器自动化与取证

浏览器自动化是 Website Analytics 系统与目标网站交互的核心能力，而反检测机制则是确保自动化流程能够顺利执行的关键保障。本章将详细阐述 Playwright MCP Server 的集成设计、反自动化检测的绕过策略，以及取证数据的采集与组织方式。

## 4.1 Playwright 集成设计

系统采用 Playwright MCP Server 作为浏览器自动化基础设施。MCP（Model Context Protocol）是一种标准化的协议规范，定义了 AI Agent 与浏览器之间的通信接口。通过 MCP Server，AI Agent 可以调用标准化的工具函数控制浏览器行为，包括页面导航、元素定位、表单填写、截图录制等操作。

系统在 MCP Server 的基础上封装了 `AutoSwitchingPlaywrightServer` 类，提供了多项增强功能以适配复杂的自动化场景。自动标签页切换功能解决了多页面操作中的上下文管理问题：当 Agent 的操作导致新标签页打开时（例如点击链接），系统会自动检测到标签页变化，并将执行上下文切换到新打开的标签页，确保后续操作作用于正确的页面。浏览器优雅关闭功能则实现了分层次的资源释放：首先调用 `browser_close` 尝试正常关闭浏览器，若失败则逐个关闭标签页，最后通过进程扫描强制终止可能残留的孤儿 Chrome 进程。这种分层清理策略有效解决了浏览器进程残留问题，特别是在任务被取消或异常中断的场景下。

视频录制功能为任务执行过程提供了完整的操作记录。系统通过配置 Playwright 的 `recordVideo` 参数，在任务执行期间自动录制浏览器操作视频。视频文件以 WebM 格式保存，以时间戳命名，便于后续的审计和问题复现。截图存储采用分层目录结构：任务根目录存放执行日志和最终报告，captures 子目录存放执行过程中的页面截图，evidence 子目录存放取证任务生成的条目截图，videos 子目录存放录制视频。这种结构化的存储方式使得任务执行的所有产物都可以被方便地检索和关联。

Cookie 持久化功能允许在任务之间复用登录状态。系统支持从指定路径加载 Cookie 文件，在浏览器启动时注入，从而跳过登录流程直接访问需要认证的页面。这一功能在需要多次访问同一网站的场景下显著提高了执行效率。

## 4.2 反检测机制

现代网站普遍采用各种技术检测和阻止自动化访问，其中 Cloudflare 的人机验证是最常见的反自动化手段之一。系统实现了专门的 Cloudflare 绕过工具，通过获取有效的 `cf_clearance` Cookie 来通过验证。

Cloudflare 绕过工具的工作流程如下：首先使用 undetected-chromedriver 启动一个能够绕过检测的浏览器实例，访问目标网站并通过人机验证；然后提取浏览器中的 `cf_clearance` Cookie 和 User-Agent 信息；最后将这些信息注入到 Playwright 控制的浏览器实例中。为了验证 Cookie 的有效性，系统使用 curl 命令发送测试请求，检查返回的 HTTP 状态码是否为 200。只有通过验证的 Cookie 才会被用于后续的自动化操作。

除了 Cloudflare 绕过，系统还实现了反检测 JavaScript 代码注入。这段代码通过修改浏览器环境中的特征属性来隐藏自动化痕迹，包括：将 `navigator.webdriver` 属性设为 `undefined`，删除 `navigator.__proto__.webdriver` 属性，添加伪造的 `chrome.runtime` 对象，配置合理的插件列表（PDF 查看器）和语言偏好（中文优先）。这些修改使得浏览器指纹更接近真实用户的环境，降低了被反自动化系统识别的概率。

人类行为模拟策略则通过随机化操作参数来避免过于规律的自动化特征。系统在页面操作之间引入随机延迟，模拟人类阅读和思考的时间；鼠标移动轨迹采用贝塞尔曲线而非直线，模拟自然的手部运动；滚动操作使用渐进式滚动而非一次性跳转。这些细节虽然微小，但能够有效降低被行为分析系统标记为机器人的风险。

## 4.3 核心业务功能

Website Analytics 系统围绕网站自动化分析场景，提供了三项核心业务功能：注册取证、订阅链接提取和订阅支付。这三项功能分别由 RegisterAgent、ExtractAgent 和 PaymentAgent 三个专业代理独立执行，通过 CoordinatorAgent 按需编排调用。

注册取证功能实现了网站账号的自动化注册与凭证留存。RegisterAgent 访问目标网站的注册页面，填写账号、密码等注册信息，触发邮箱验证码发送后，通过 IMAP 协议从邮件服务器获取验证码并完成验证。注册成功后，系统保存新账号的登录凭证（账号名、密码）以及注册过程中的关键页面截图，形成完整的注册取证记录。该功能采用严格的工具白名单机制，仅允许使用五个基础浏览器操作工具，确保注册过程的安全可控。

订阅链接提取功能用于在已登录状态下获取用户的订阅地址。ExtractAgent 导航到网站的用户中心或订阅管理页面，定位订阅链接的展示位置或复制按钮，提取完整的订阅 URL 地址。系统会对提取的链接进行有效性验证，确保返回的订阅地址格式正确且可正常访问。该功能支持在完成登录或注册后自动触发，实现从认证到数据提取的完整闭环。

订阅支付功能专注于支付二维码的自动提取。PaymentAgent 在已登录状态下访问网站的订阅购买页面，选择目标套餐后进入支付流程，定位并截取支付二维码图像。系统要求保存三张关键截图（订阅页面、支付方式选择页、二维码展示页），同时尝试解析二维码图像提取支付链接或支付码文本。提取的支付凭证将随任务结果一并上报，供下游系统完成支付验证或记录存档。

三项功能的执行结果均以结构化的方式保存，包括 JSON 格式的元数据、PNG 格式的截图证据以及操作过程录屏，形成可追溯的完整证据链。

# 第五章 核心流程实现

本章将深入系统的代码实现层面，阐述主执行流程的编排逻辑、输出类型的设计原则以及关键代码的实现细节。这些内容对于理解系统的运行机制和技术细节具有重要价值。

## 5.1 主执行流程

`orchestrator.py` 模块是系统的主执行入口，负责协调各个子系统的协作，完成从任务初始化到结果返回的完整流程。主执行流程 `execute()` 函数采用异步编程模型，通过上下文管理器确保资源的正确分配和释放。

执行流程始于任务环境的初始化。系统根据当前时间生成唯一的任务目录，创建日志记录器，并初始化 LLM 调用钩子（Hooks）用于捕获和记录 AI 模型的调用过程。接下来，系统构建 Playwright 启动参数，包括无头模式配置、代理服务器设置和视频录制路径等。Playwright MCP Server 以异步上下文管理器的形式启动，确保在任务结束时自动执行浏览器清理。

在 Agent 构建阶段，系统从指令文件加载各代理的提示词（Prompt），并根据任务配置创建五个专业代理。LoginAgent 负责登录操作，RegisterAgent 负责账号注册（绑定邮箱验证码获取工具），ExtractAgent 负责订阅数据提取，EvidenceAgent 负责网站取证（绑定截图和报告生成工具），PaymentAgent 负责支付二维码提取。这五个子代理通过 `as_tool()` 方法被封装为可调用的工具函数，供 CoordinatorAgent 调度使用。

CoordinatorAgent 是执行流程的中央协调者。它接收用户指令，解析任务意图，按照逻辑顺序调用相应的子代理工具。例如，对于"登录并提取订阅"的指令，CoordinatorAgent 会先调用 LoginAgent 完成登录，确认登录成功后再调用 ExtractAgent 提取订阅地址。每个子代理的调用都受限于最大轮数（max_turns），防止无限循环。执行完成后，CoordinatorAgent 汇总各子代理的输出，生成统一的结构化结果。

主执行流程的异常处理采用防御性编程策略。任何阶段的异常都会被捕获，并转换为标准的失败输出格式。即使在异常情况下，系统也会尽力收集已产生的执行数据（如截图、视频路径），确保故障可追溯。执行结果封装为 `ExecutionResult` 对象，包含成功标志、退出码、任务目录、协调代理输出和 LLM 使用统计等字段，为调用方提供完整的执行上下文。

## 5.2 输出类型设计

系统的输出类型设计基于 Pydantic 框架，实现了严格的结构化数据校验。这种设计确保了 AI Agent 的输出符合预期的格式，便于下游系统解析和处理。

错误类型体系（ErrorType）是输出设计的核心组成部分。系统定义了涵盖多种失败场景的错误类型枚举，按照业务优先级分为六大类别。账号/套餐类错误包括账号被封禁（ACCOUNT_BANNED）和订阅套餐失效（PLAN_EXPIRED），这类错误表明任务失败源于账号状态问题。网站访问类错误包括服务器错误、网络错误、域名错误以及登录/注册页面未找到，反映了目标网站的可访问性问题。反自动化类错误包括反自动化检测（ANTI_AUTOMATION_DETECTED）和人机验证失败（HUMAN_VERIFICATION_FAILED），表明任务被网站的安全机制拦截。业务流程类错误包括复制按钮未找到、订阅地址异常和邮箱验证失败，这类错误与特定业务逻辑的执行失败相关。任务限制类错误包括任务超时（TASK_TIMEOUT）和任务步骤超限（TASK_STEP_LIMIT），表明任务因资源或时间限制而终止。最后是运行时错误类，包括任务被清理（TASK_CLEANED）和未知错误（UNKNOWN_ERROR）。

各子代理的输出类型继承自 Pydantic 的 `BaseModel`，统一包含 success、message 和 error_type 三个核心字段。LoginOutput 包含登录操作的结果状态；RegisterOutput 额外包含注册使用的账号和密码；ExtractOutput 包含提取到的订阅地址；EvidenceOutput 包含取证的入口统计信息和报告文件路径；PaymentOutput 包含支付码信息和多张截图的路径。CoordinatorOutput 作为最终输出，汇总了所有子代理的执行结果，包含任务整体状态、详细消息、错误类型、执行的操作列表以及各操作的完整结果字典。

这种层次化的输出设计使得系统能够精确地描述执行结果。不仅知道任务是否成功，还能知道失败的具体原因和阶段，为问题诊断和重试策略制定提供了丰富的信息。

## 5.3 关键代码实现

工具过滤器 `_playwright_tool_filter` 是系统安全控制的核心实现。该函数在每次工具调用前执行权限检查，采用双层过滤策略。全局黑名单层屏蔽了 `browser_handle_dialog` 和 `browser_run_code` 两个危险工具：前者无法处理页面内的 DOM 弹窗，仅支持原生对话框；后者允许执行任意 JavaScript 代码，存在安全风险。Agent 级别白名单层针对 RegisterAgent 实施严格限制，仅允许使用五个核心浏览器操作工具（browser_navigate、browser_snapshot、browser_click、browser_fill_form、browser_press_key），防止注册代理执行非预期的危险操作。其他代理则放行所有非黑名单工具，获得更大的操作灵活性。

LLM Token 统计机制通过 `LLMUsageStats` 数据类实现。该统计器在任务执行过程中累积记录输入 Token、输出 Token、缓存 Token、推理 Token 的总消耗量，以及 LLM 调用次数。统计数据通过自定义的 `RunHooks` 实现捕获：在每次模型调用的响应回调中，从使用信息对象（Usage）中提取 Token 数量并累加到统计器中。任务完成后，统计数据被转换为字典格式，随执行结果一并返回，为成本核算和性能优化提供数据支撑。

任务结果组装逻辑在 `_save_single_task_summary` 函数中实现。该函数收集任务执行过程中的所有关键信息，包括任务标识、指令内容、执行时长、任务目录、协调代理输出、退出码、起止时间和 LLM 使用统计，封装为 `TaskResult` 对象并序列化为 JSON 文件保存。这种详细的任务总结文件便于后续的批量分析和报告生成，也是系统可追溯性的重要保障。

系统的模块化设计和清晰的分层架构使得各组件可以独立演进和测试。浏览器驱动层可以适配不同的 MCP Server 实现，AI Agent 层可以切换不同的 LLM 提供商，任务调度层可以扩展新的任务类型，而业务逻辑代码无需大幅修改。这种可扩展性为系统的长期演进奠定了坚实的技术基础。
